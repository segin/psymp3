### 1. Unsigned underflow of first-partition sample count causes heap buffer overflow
**`src/codecs/flac/ResidualDecoder.cpp:87`** · group: codecs-flac-b

decodeResidual computes samples_per_partition = block_size / partition_count and then sets the first partition's count to `samples_per_partition - predictor_order` (line 87) with no check that samples_per_partition >= predictor_order. partition_order is an attacker-controlled 4-bit field (parseResidualHeader), so partition_count can be up to 32768. With e.g. block_size=4096, partition_order=10 (partition_count=1024, samples_per_partition=4) and an LPC predictor_order=8, the subtraction underflows to ~4.29e9 (uint32). decodePartition then loops that many times writing output[i] into the `residuals` buffer that SubframeDecoder allocated with only `block_size - order` ints, producing a massive out-of-bounds heap write driven directly by a crafted FLAC frame. The output_offset != residual_count check at line 131 happens only after the overflow. Add a check that block_size/partition_count > predictor_order (and that partition layout is valid) before decoding.

> **Verifier:** Traced the full path. SubframeDecoder::decodeLPC allocates `residuals = new int32_t[block_size - order]` (SubframeDecoder.cpp:382) where `order` is a file-derived LPC predictor order (1..32, validated at :303) and passes it to decodeResidual. Inside decodeResidual, `partition_order` is read directly from the bitstream (parseResidualHeader, ResidualDecoder.cpp:166) and the ONLY validation is `block_size % partition_count != 0` (line 67); there is no check that `samples_per_partition >= predictor_order`. With e.g. block_size=4096, order=8, partition_order=10 (partition_count=1024, 4096%1024==0 passes), samples_per_partition=4, so line 87 computes `4 - 8` which underflows to 0xFFFFFFFC (~4.29e9). decodePartition (line 123/181) then writes `output[i]=value` for each successfully Rice-decoded sample with no per-index bounds check; a frame padded with set bits yields quotient=0/value=0 (passes the quotient<=1024 and isValidResidual guards), so each iteration writes past the 4088-int heap buffer until the bitstream is exhausted. The output_offset!=residual_count check (line 131) runs only after the overflow. Concrete, attacker-controlled heap buffer overflow.


## HIGH (18)


### 2. Decoder buffer grows without bound while playback is paused
**`src/audio.cpp:322`** · group: app-toplevel

The decoder thread's back-pressure wait uses the predicate `m_buffer.size() < BUFFER_HIGH_WATER_MARK || !m_active || !m_playing`. When the user pauses (Player calls audio->play(false), player.cpp:831, setting m_playing=false), the predicate is always true, so the wait returns immediately and the loop keeps calling getData and appending 4096-sample chunks to m_buffer (line 391) with no size check; meanwhile SDL_PauseAudioDevice stops the callback so nothing drains the buffer. The decoder therefore decodes the entire remaining file into RAM as raw PCM — e.g. pausing a 1-hour 48 kHz stereo track accumulates roughly 690 MB, and multi-hour files reach gigabytes. The `!m_playing` term was intended only to wake the thread for shutdown but defeats the high-water mark entirely.

> **Verifier:** Confirmed by direct trace: Player::pause() (src/player.cpp:831) calls Audio::play(false) (src/audio.cpp:170-173), which sets m_playing=false and pauses the SDL device, stopping the only buffer drain (m_buffer.erase, src/audio.cpp:451). The decoder thread's wait predicate (src/audio.cpp:321-323) includes `|| !m_playing`, so while paused it never blocks on the high-water mark, and the inner loop appends decoded chunks unconditionally at src/audio.cpp:391 with no other size check until EOF — decoding the entire remaining file into m_buffer as raw PCM (~690 MB/hour at 48 kHz stereo). Triggers on every pause; impact ranges from tens of MB and a CPU/disk burst for normal tracks to gigabytes for multi-hour files.


### 3. Mid-side decorrelation reconstructs wrong samples for odd side values
**`src/codecs/flac/ChannelDecorrelator.cpp:148`** · group: codecs-flac-b

decorrelateMidSide computes side_half = side >> 1, left = mid + side_half, right = mid - side_half, then for odd side decrements/increments only the right channel. RFC 9639 / libFLAC require restoring the dropped LSB into the mid before reconstruction: mid2 = (mid<<1)|(side&1); left=(mid2+side)>>1; right=(mid2-side)>>1. For L=5,R=2 the encoder stores mid=3, side=3; the reference yields left=5,right=2, but this code yields left=4 (line 148) and right=1, i.e. both channels are off by one for every odd-side sample. Since the warm decoded mid channel (ch 0) is passed raw (NativeFLACCodec sets is_side_channel only for ch 1) the LSB is never restored, so any mid-side stereo frame — the most common FLAC encoding — with odd side samples is decoded with off-by-one amplitude errors, producing audible distortion. The code's own comment ('left+right = 2*mid - 1 for odd positive side') contradicts the spec value of 2*mid + 1.

> **Verifier:** Confirmed. ChannelDecorrelator.cpp:138-174 computes left = mid + (side>>1) and right = mid - (side>>1) with an odd-side adjustment only to right, instead of the RFC 9639/libFLAC reconstruction mid2=(mid<<1)|(side&1); left=(mid2+side)>>1; right=(mid2-side)>>1 — for mid=3, side=3 it yields left=4/right=1 instead of 5/2 (the comment at line 162 even asserts left+right=2*mid-1, contradicting the spec invariant L+R=2*mid+(side&1)). The path is live: "flac" is registered to the native FLACCodec (src/codecs/CodecRegistration.cpp:107-108, include/codecs/flac/NativeFLACCodec.h:113), decorrelate() is called per frame at src/codecs/flac/NativeFLACCodec.cpp:577, and nothing else restores the mid LSB (is_side_channel set only for ch 1 at NativeFLACCodec.cpp:545; SubframeDecoder.cpp:166 only widens the side channel). Trigger: any mid-side stereo frame with odd side samples — i.e., ~half the samples of virtually every real-world stereo FLAC — decodes with ±1 errors in both channels, breaking lossless reconstruction; perceptual impact is LSB-level noise rather than gross distortion, but it is a near-universal correctness failure of the decoder's core function.

### 6. Unsigned underflow in next-sync search produces heap out-of-bounds read
**`src/demuxer/flac/FLACDemuxer.cpp:524`** · group: demuxer-flac

In readChunk_unlocked the outer loop `while (bytes_read > search_start)` (search_start defaults to MIN_FRAME_SIZE=9, or min_frame_size) is entered whenever bytes_read exceeds search_start. > **Verifier:** The path is reachable and the underflow is genuine. `search_start` stays at MIN_FRAME_SIZE=9 whenever STREAMINFO is invalid or `min_frame_size` is 0 — and per the struct (FLACDemuxer.h:72 "0 if unknown") that is a common, RFC-legal case. `calculateFrameSize_unlocked` floors estimates at 14 (FLACDemuxer.cpp:3798-3802, 3853), but for a truncated final frame `available_bytes` is reset to `m_file_size - frame.file_offset` (line 452) and the only EOF guard rejects `< 9` (line 456), so `available_bytes`/`bytes_read` can legitimately be 10-15. The outer loop at line 521 is then entered (`bytes_read > 9`), and `for (size_t i = search_start; i < bytes_read - 16; ++i)` (line 524) computes `bytes_read - 16` which underflows to a near-2^64 size_t, so the loop never terminates on bound and `data[i]`/`data[i+1]`/`data[i+2]` (lines 525,529) read past `data.size()==available_bytes`; `parseFrameHeader_unlocked(data.data()+i, bytes_read - i)` (line 537) also gets an underflowed huge length once `i > bytes_read`. Concrete trigger: a FLAC file with STREAMINFO min_frame_size=0 truncated so the last frame offset leaves 10-15 trailing bytes reliably produces a heap out-of-bounds read and near-certain crash.


### 7. Integer overflow in stsz size check enables unbounded allocation
**`src/demuxer/iso/BoxParser.cpp:940`** · group: demuxer-iso-a

ParseSampleSizeBox reads sampleCount (uint32_t) straight from the file (line 925) then validates `size < 12 + (sampleCount * 4)`. The code then does `tables.sampleSizes.reserve(sampleCount)` (line 944) and loops `sampleCount` times calling ReadUInt32BE, attempting a multi-gigabyte allocation and ~1e9 iterations. > **Verifier:** The bug is real and triggerable. The code then calls `tables.sampleSizes.reserve(sampleCount)` (line 944, ~4GB) and loops `sampleCount` times (~1 billion) — and unlike ParseTimeToSampleBox (lines 859-860) there is no MAX_SAMPLES_PER_TRACK clamp here. ### 8. Integer overflow in stco/co64 size check enables unbounded allocation
